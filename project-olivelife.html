1. 주제 선정 배경

4인 팀으로 진행한 뷰티 커머스 백엔드 프로젝트입니다. 대규모 트래픽 검증 환경 구축을 담당했으며, 트래픽 증가 상황에서 발생할 수 있는 병목과 장애를 사전에 검증하는 데 집중했습니다. Multi-Level Cache 아키텍처와 분산 부하 테스트 환경을 구축하여 검색 성능을 개선하고, 대규모 요청 환경에서도 안정적으로 동작할 수 있는 기반을 마련했습니다.

2. 담당 역할

  • 상품 검색 성능 개선 및 대규모 트래픽 검증 환경 구축 담당
  • k6 부하 테스트 기반 병목 분석 및 성능 개선 주도
  • 캐시 아키텍처 설계와 데이터 일관성 보장 전략 구현

6. 기술적 의사결정 및 트러블슈팅

k6 부하 테스트 설계 및 다단계 캐시(Multi-Level Cache) 성능 검증

배경

사전 부하 테스트 시 100 VUs부터 심각한 성능 저하가 발생하는 것을 확인했습니다. 이에 서버 한계를 상회하는 부하를 가해 Breaking Point를 특정하고, 장애 상황을 재현하여 근본적인 병목 지점을 찾기 위한 체계적인 부하 테스트가 필요했습니다.

기술적 의사결정

Docker 기반으로 k6(부하 생성)와 InfluxDB, Grafana(실시간 모니터링)를 연동하여 멱등성 있는 테스트 환경을 구축했습니다. 또한, 단순한 트래픽 주입이 아닌 30s(Ramp-up) → 1m(Steady) → 30s(Step-up)의 단계별 부하 증가 시나리오(목표 150 VUs)를 설계하여 단계별 캐시 전략(v1~v3)의 성능을 정밀하게 비교 분석하기로 결정했습니다.

주요 구현

  • 부하 테스트 환경 구축: k6, Docker, InfluxDB, Grafana를 활용한 실시간 성능 모니터링 체계 구축
  • 테스트 시나리오 설계: 에러율 5% 미만, p(95) 응답속도 500ms 이내를 목표로 점진적 부하 증가 전략 적용
  • v1(No Cache) 병목 분석: 타임아웃(dial: i/o timeout) 발생 및 상위 5% 사용자 11초 이상 대기 현상 확인. 느린 DB 조회가 연결 자원을 모두 점유해 후순위 요청이 실패하는 '자원 고갈' 원인 도출
  • 캐시 전략별 성능 비교: v1(No Cache), v2(Local Cache), v3(Redis Cache) 환경에서의 평균 응답 시간, p(95) 응답 시간, TPS, 에러율 비교 검증

성과

캐시 미적용 시 발생하던 2.75%의 에러율을 0.00%로 완벽하게 해결했습니다. p(95) 응답 시간을 11.68초에서 약 112.14ms(v3 기준)로 획기적으로 단축했습니다. v3(Redis Cache) 도입을 통해 TPS를 16.27에서 75.96으로 약 4.6배 향상시켰으며, 분산 환경에서의 데이터 정합성과 확장성(Scale-out)을 모두 보장하는 최적의 아키텍처를 도출했습니다.

다단계 캐싱(Multi-Level Cache)과 Redis Pub/Sub을 활용한 검색 성능 및 정합성 개선

배경

동일한 키워드의 반복적인 검색 요청이 매번 DB 조회로 이어져, 트래픽 증가 시 심각한 DB 부하와 응답 지연이 발생했습니다. 이를 해결하고자 캐시를 도입했으나, 다중 인스턴스(Scale-Out) 분산 환경에서 특정 서버에만 과거 데이터가 남아 서로 다른 응답을 반환하는 심각한 '캐시 데이터 불일치(Inconsistency)' 문제가 발견되었습니다. (예: APP1은 변경된 재고 250을 반환하지만, APP2는 옛날 재고 100을 반환하는 현상)

기술적 의사결정

DB 조회를 최소화하고 성능을 극대화하기 위해 캐시 아키텍처를 4단계(v1~v4)로 고도화했습니다. 최종적으로 Caffeine(L1) + Redis(L2)를 결합한 다단계(Multi-Level) 캐시 구조를 채택하여 네트워크 비용과 응답 속도를 개선했습니다. 또한, 분산 환경에서의 고질적인 데이터 정합성 문제를 완전히 해결하기 위해 Redis Pub/Sub 기반의 캐시 무효화(Invalidation) 동기화 전략을 도입하기로 결정했습니다.

주요 구현

  • 단계별 캐시 고도화 설계: v1(DB 직접 조회) → v2(Caffeine 로컬 캐시) → v3(Redis 분산 캐시) → v4(L1+L2 2단계 캐시 및 Pub/Sub 동기화)로 점진적 아키텍처 개선
  • 분산 환경 정합성 병목 분석: 데이터 갱신 시, 요청을 처리한 특정 인스턴스의 캐시만 삭제되어 나머지 인스턴스들은 변경 이전의 낡은 값을 반환하는 원인 파악 및 시뮬레이션
  • Redis Pub/Sub 캐시 동기화 로직: 데이터 변경 발생 시 Redis Pub/Sub을 통해 모든 애플리케이션 인스턴스에 '캐시 삭제' 이벤트를 즉각 발행
  • 라우팅(Nginx)에 의해 어느 인스턴스로 요청이 인입되더라도 항상 DB의 최신 갱신 값과 일치하는 응답을 반환하도록 처리

성과

반복적인 DB 조회를 효과적으로 차단하여 검색 API의 응답 지연(병목)을 완벽히 해소했습니다. Redis Pub/Sub 기반 동기화를 통해 다중 서버 환경에서 발생하던 데이터 불일치 문제를 원천 차단하여 100%의 캐시 정합성을 확보했습니다. 결과적으로 시스템의 확장성(Scale-out)을 제약 없이 보장하면서도, 사용자에게 빠르고 정확한 실시간 검색 경험을 안정적으로 제공할 수 있게 되었습니다.

복합 인덱스 튜닝 및 Redis ZSet 기반 검색 인프라 구축

배경

대규모 상품 데이터 환경에서 카테고리, 상품 상태, 상품명 조건 및 정렬이 포함된 검색 쿼리 실행 시 Filesort가 발생하며 응답 속도가 저하되는 문제를 확인했습니다. 또한 사용자의 검색 데이터를 활용한 검색 경험 개선이 필요했습니다.

기술적 의사결정

실행 계획(EXPLAIN)을 분석하여 실제 검색 패턴에 최적화된 복합 인덱스(category_id, status, name)를 설계했습니다. 또한 실시간 인기 검색어 기능 구현을 위해 정렬 및 랭킹 조회에 강점을 가진 Redis Sorted Set(ZSet)을 활용했습니다.

주요 구현

  • EXPLAIN 기반 쿼리 실행 계획 분석 및 병목 구간 파악
  • 복합 인덱스(category_id, status, name) 설계 및 적용
  • Redis ZSet 기반 실시간 인기 검색어 기능 및 IP + TTL 기반 중복 검색어 카운트 방지 로직 적용

성과

검색 쿼리 성능 개선 및 Filesort 제거 성공. 실시간 인기 검색어 기능을 제공하여 사용자 검색 데이터 기반 탐색 경험을 크게 향상시켰으며, 데이터 증가 상황에서도 안정적인 검색 성능을 확보했습니다.
